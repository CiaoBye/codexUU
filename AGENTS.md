# CodexUU 协作规范

默认使用中文回复。修改本项目时遵循以下规则：

1. 先理解目标，再进行小步、可回滚的修改；避免无关重构和随意新增依赖。
2. 涉及代码时说明核心逻辑、影响范围和验收方式。
3. 发现问题时区分“必须修复”“建议优化”“可选提升”。
4. 每次完成代码更新后，必须同步更新本文件和 `agents/changelog.md`，记录本次改动、影响范围和验证结果。
5. 每次完成更新后必须重启程序；若程序未运行，则启动一次进行验证。
6. 参考上游 `shanggqm/codexU` 的 README、CHANGELOG 和 Release notes，优先避免数据口径、兼容性、刷新和更新检查方面的潜在问题。
7. macOS 专属功能除非明确要求，否则不作为当前跟进重点。
8. Windows 主界面固定采用 codexU 式单页仪表盘：顶部为全局操作，上半区常驻额度、今日/本周/本月/累计 token 与本月羊毛进度，下半区仅通过“今日任务 / 用量趋势 / 项目排行 / Skill”Tab 原地切换；禁止恢复左侧页面导航或把概览拆成独立页面。
9. 功能复刻必须同时检查数据来源、空状态、加载状态、错误状态、hover 详情和浅色主题可读性，不能只验证默认截图。
10. 当前 Windows 版本的功能验收范围包括：后台 Runtime、额度窗口、token 拆分、任务、每日/每周/每月/累计趋势、项目本周/本月/累计口径、真实工具调用、Skill 加载和更新检查。
11. 同一 Runtime 的概览、趋势和项目统计必须来自同一精细数据口径；若任意页面数值不一致，不得交付。
    - “近 7 天”必须表示连续 7 个自然日，缺失日期按 0 展示，不能替换为最近 7 个活跃日。
    - 累计 token 优先显示 `state_5.sqlite` 的线程索引总量，输入/缓存/输出拆分来自 session `token_count`，两种来源必须在 UI 中可解释。
    - “本月羊毛进度”只能使用所选统计时区下的当月精细 token，不得误用累计 token。
12. UI 交付前必须用真实本机数据检查所有主页面，禁止只用空状态或构造数据截图作为完成证据。
13. 重启程序时必须关闭所有指向本项目 `main.py` 的旧 `pythonw.exe` 实例，再用绝对路径启动，并确认最终实例数严格为 1。
14. 工具榜只能统计明确的 `function_call` / `custom_tool_call` 或对应 Runtime 的显式工具事件；不得按调用次数把总 token 或总金额比例分摊到单个工具。Skill 只统计明确的 Skill 加载记录，不把“已安装”或普通文本提及算作使用。
15. Codex API 等效价值必须按日志中的实际 OpenAI 模型使用官网公开的输入、缓存输入和输出单价分别计算；未知、内部或第三方模型必须标记为未计价，并显示计价覆盖率，禁止借用其他模型价格。
16. 顶部 Runtime 不在主界面展示，统一放入设置页；主界面顶部保留主题、中文/English、刷新、设置和关闭控件。没有真实账号状态的数据徽章和低价值控件不得常驻。
17. 所有累计、趋势和项目数字必须明确是“本机记录”，不得冒充 Codex 账号云端活动页统计；除非未来接入官方账号级接口，否则不承诺与跨设备、云端任务或已清理历史数据一致。
18. 项目排行只统计仍存在的真实项目目录；排除已删除目录、日期型一次性会话工作区、Temp/AppData/`.codex` 等内部目录。创作类项目可无代码清单，但目录必须存在且非空。
19. “本周”按所选统计时区的周一 00:00 至周日 23:59 计算；不得用滚动近 7 天代替。
20. 产品对外可见名称统一为 `CodexUU`；为兼容既有用户设置，内部 `.codexU` 配置目录和上游仓库标识不得随意迁移。
21. 趋势页统计口径按钮固定左对齐；左侧“Token 活动 + 当前范围数据”和右侧趋势标题必须位于各自卡片内部，并保持两张卡片标题栏、内容区和总高度一致。
22. 额度窗口必须按后端实际返回的窗口动态渲染；缺失的 5h/7d 窗口不得显示占位数字、空环或重置行。重置时间使用所选统计时区，5h 显示时分，7d 显示月日与时分，hover 可查看含秒的完整时间。
23. 主 Tab 切换采用短时滑块与内容淡入位移动效，趋势口径采用轻量交叉淡化；动效不得阻塞数据刷新。顶栏和指标图标优先使用项目 SVG 资源，避免依赖不同系统显示不一致的 Unicode 符号。
24. 每日 Token 热力图的最后一行必须与右侧趋势图的 X 轴基线对齐，不得在热力图下方遗留额外空白。项目排行采用“左侧项目用量排行 + 右侧活动概览”双栏结构；概览至少包含活跃项目、本期新增、Top1/Top3 集中度与最近活跃项目，且全部指标跟随本周/本月/累计口径同步切换。
25. 视觉系统使用统一设计令牌：指标数字优先使用 Segoe UI Variable，未缓存/缓存/输出分别使用蓝/紫/橙语义色，主背景采用克制的环境光渐变，核心卡片提供轻量材质、阴影、趋势徽章和数值缓动；必须提供“减少动态效果”设置。
26. Windows 产品化设置必须保持可用：额度已用/剩余口径、原生全局快捷键、托盘快速状态悬浮窗、GitHub Release 页面与下载入口、数据源诊断、刷新反馈、主窗口置顶和关闭行为都需要持久化或即时生效。
27. 项目版本从 `0.0.1` 起步，以根目录 `VERSION` 为单一版本来源，`app/constants.py` 只负责运行时读取。版本按十轮分段进位：`0.0.1 … 0.0.10 → 0.1.01 … 0.1.10 → 0.2.01`；每完成一轮代码更新递增一次，并同步 README、本文件和 `agents/changelog.md`。发布 Git 标签统一使用 `v<版本号>`。
28. Windows 11 状态栏采用系统通知区动态图标、tooltip 和点击浮窗实现，不注入系统不支持的常驻任意文本。自动全量刷新默认 60 秒并使用粗粒度定时器与刷新互斥；托盘状态图标仅在可见状态变化时重绘，避免空闲 CPU 回环。
29. 顶部指标卡不得常驻环比/计价覆盖等额外徽章；未缓存、缓存、输出必须保持同一行。每日热力图需要同时利用横纵空间，上下绘图区与右侧趋势图协调对齐。
30. 今日任务“完成”以线程今天被归档为准，优先读取 `archived_at`，旧 SQLite 结构才回退 `updated_at`；未归档线程近 2 小时有活动为“进行中”，今天更早有活动为“待处理”。模型停止输出不能直接视为完成。
31. 手动和定时刷新必须失效聚合快照，但应保留有上限的文件级解析缓存；Codex rollout 文件缓存最多保留 1024 项。SQLite 数据源需同时兼容根目录与 `.codex/sqlite/` 布局，并统一使用只读连接和短超时。
32. 全局快捷键变更若注册失败，必须恢复上一个已生效组合键；后台更新检查完成后只能通过 Qt Signal 回到主线程更新 UI。通知区图标位置不可用时，快速状态面板回退到当前屏幕右下角。
33. Claude 增量索引仅可保存文件指纹和派生统计（时间、模型、Token、工具、Skill）；不得复制或持久化 transcript 正文、提示词、工具参数或项目文件内容。索引损坏或删除时必须可从原始日志安全重建。
34. 日常 GitHub 推送和版本标签不得自动构建 Release；只有用户明确指定后才能在 GitHub Actions 手动触发 Windows 构建或发布。发布时必须产出可安装 `.exe`、SHA-256 文件和 Release 下载资产；构建脚本不得依赖用户工作目录中的未提交文件。
35. 项目详情只能从当前项目的精细 Token 事件派生模型与会话统计；导出必须注明当前统计口径，且不得包含 transcript 正文、提示词、工具参数或原始项目文件内容。
36. 额度提醒默认按“剩余比例”判断，与界面已用/剩余显示口径无关；同一 Runtime、窗口和重置周期最多提醒一次。数据维护只能清理可从原始日志重建的派生索引及 SQLite 辅助文件，严禁清理原始日志。
37. 桌面状态悬浮窗默认显示但允许在设置和托盘菜单即时隐藏；只展示当前 Runtime 的本机 Token、实际额度与重置时间。窗口可拖动并持久化位置，恢复时必须按当前可用屏幕边界校正，避免多显示器变化后落到屏幕外。

推荐使用 `powershell -ExecutionPolicy Bypass -File scripts/restart.ps1` 完成重启和单实例校验。

## 验收要求

- Python 修改：运行 `python -m compileall -q app main.py`。
- 有测试时运行 `PYTHONPATH=. python -m pytest -q`。
- Qt 界面修改：使用 offscreen Qt smoke test 验证窗口和关键控件可创建。
- 更新完成后确认程序已经重启，并在 `agents/changelog.md` 留下记录。
